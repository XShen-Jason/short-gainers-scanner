# =========================
# 项目名称：Short Gainers Screener
# =========================

你正在开发一个用于加密货币“空涨幅榜分析”的数据采集与处理系统。

系统目标：
- 从 OKX 获取市场数据
- 结构化整理 K线与涨幅榜数据
- 计算基础技术指标
- 提供统一 JSON 输出
- 为后续 AI 分析提供输入数据

⚠️ 本系统不做交易决策，只做数据层与基础处理。

# =========================
# 一、架构约束
# =========================

必须遵循分层架构：

1. data-fetcher（数据获取层）
2. data-processor（指标计算层）
3. storage-layer（数据库层）
4. api-layer（查询接口层）

禁止：
- 将所有逻辑写在单一文件
- 将数据库操作与业务逻辑混合
- 在 API 层直接写数据抓取逻辑

# =========================
# 二、数据来源
# =========================

数据必须来源于：
- OKX Public API

禁止：
- 使用模拟数据（除非用于测试）
- 使用非官方数据源

# =========================
# 三、数据范围
# =========================

必须支持：

(1) 涨幅榜数据：
- symbol
- price
- 24h change %
- 24h high / low
- 24h volume
- rank

(2) K线数据：
周期必须支持：
- 5m
- 15m
- 1h
- 4h
- 1d

每条K线必须包含：
- timestamp
- open
- high
- low
- close
- volume

(3) 衍生指标：
至少实现：
- rolling high
- rolling low

可选：
- MA / RSI / EMA

# =========================
# 四、数据处理规则
# =========================

1. 所有数据必须标准化为统一 JSON 结构
2. 不同 symbol 结构必须一致
3. 所有时间字段使用 timestamp（UTC）

rolling high/low：
- 使用固定窗口（例如最近 N 根K线）
- 每个周期独立计算

禁止：
- 使用不一致的数据格式
- 混用不同时间单位

# =========================
# 五、性能与请求控制
# =========================

必须实现：

- 并发控制（Promise pool / queue）
- 请求限速（rate limiting）
- 批量处理 symbol

禁止：
- 无限制并发请求
- 每个 symbol 串行请求多个周期（低效实现）

# =========================
# 六、存储规范
# =========================

必须使用外部数据库（推荐 PostgreSQL / Supabase）

存储内容包括：
- symbol
- K线数据
- 涨幅榜数据
- 指标数据

禁止：
- 使用 Worker KV 作为主存储
- 将所有数据存入内存

KV 仅可用于：
- 缓存
- 临时数据

# =========================
# 七、API 设计规则
# =========================

API 必须提供：

- 获取涨幅榜
- 获取某 symbol K线
- 获取处理后的结构化数据

API 返回必须：
- JSON 格式
- 字段稳定
- 可被 AI 直接消费

# =========================
# 八、代码规范
# =========================

必须遵守：

- 模块化
- 单一职责原则
- 清晰目录结构
- 可维护性优先

每个模块必须：
- 有明确输入输出
- 避免隐式依赖

# =========================
# 九、错误处理
# =========================

必须实现：

- API 请求失败重试机制
- 超时处理
- 异常日志记录

禁止：
- 静默失败
- 忽略错误

# =========================
# 十、输出要求
# =========================

所有输出数据必须满足：

- 结构一致
- 字段完整
- 可扩展
- 可供 AI 直接使用

# =========================
# 十一、开发流程约束
# =========================

开发必须按照顺序：

1. 设计目录结构
2. 实现数据获取模块
3. 实现数据处理模块
4. 实现存储模块
5. 实现 API 模块
6. 提供运行示例

禁止跳步开发或混乱实现。

# =========================
# 十二、核心原则
# =========================

- 数据优先，而不是策略优先
- 结构优先，而不是功能堆叠
- 可扩展优先，而不是一次性写完
- 模块清晰优先，而不是代码简化
